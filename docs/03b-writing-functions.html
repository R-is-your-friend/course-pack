For each function, we’ve learned how we can use it to get the results we were after, but we haven’t dug into what makes a function a function. That’s what this reading is about!

5.1 Motivation

It is common in statistics and data science to need to rescale a variable before including it in a model. Rescaling is quite helpful when variables are on very different scales. In fact, for some machine learning methods (e.g., K-nearest neighbors) it is critical that every variable be on the same scale for an analysis.

Take for example, some common measures of the economy. The consumer price index (CPI) measures the yearly change in price for a representative basket of goods and services1 and is recorded as a weighted average of prices (e.g., $250). Alternatively, the PPI (producer price index) measures the yearly change in prices paid to domestic producers of goods and services (e.g., $120). Finally, the PCE (Personal Consumption Expenditures) is a slightly broader measure than CPI in that it attempts to measure both how much households are spending (e.g., $14,000) and what they’re spending their money on.

Notice how each of these variables are on different scales? Well, that’s exactly why we might want (need) to rescale them!

5.1.1 Min-Max Scaling

One common method for rescaling is called “min-max scaling.” For any numeric vector, this method finds the maximum value and the minimum value and then uses these values to rescale each value of the vector. Mathematically, each \(i^{th}\) value of a vector \(x\) is scaled based on this calculation:

\[\frac{x_i - min(x)}{max(x) - mix(x)}\] For example, if we have a vector x with values c(1, 2, 5, 7). Then the minimum of x is 1 and the maximum of x is 7. We then use these values to rescale every value of x. This would look like the following code:

x <- c(1, 2, 5, 7)

(x - min(x)) / (max(x) - min(x))
[1] 0.0000000 0.1666667 0.6666667 1.0000000

Notice that the minimum value is now 0 because 1 - 1 (the minimum is 1) is 0 and the maximum value is now 1 because (7 - 1) / (7 - 1) (the maximum is 7) is 1.

We could rescale every column of economy using the same code as above:

economy$cpi <- (economy$cpi - min(economy$cpi)) / (max(economy$cpi) - min(economy$cpi))
economy$ppi <- (economy$ppi - min(economy$ppi)) / (max(economy$ppi) - min(economy$pci))
economy$pce <- (economy$pce - min(economy$pci)) / (max(economy$pce) - min(economy$pce))

There are a couple of issues with this process. By repeating this same process multiple times, it is harder to understand what the code is doing and we’ve made our code more error prone. Did you notice the error we made?

A better method would be to write a function that allows us to implement this process anytime we want to rescale a numeric vector!

5.1.2 Don’t Repeat Yourself

The critical motivation behind writing your own functions is the “don’t repeat yourself” (DRY) principle. In general, “you should consider writing a function whenever copied and pasted your code more than twice (i.e. you now have three copies of the same code)” (Wickham & Grolemund, 2020).

One of our favorite papers, Best Practices for Scientific Computing, summarizes this idea in a slightly different way:

Anything that is repeated in two or more places is more difficult to maintain. Every time a change or correction is made, multiple locations must be updated, which increases the chance of errors and inconsistencies. To avoid this, programmers follow the DRY (Don’t Repeat Yourself) Principle.

The DRY Principle advocates that you modularize your code—break it into smaller, independent, and reusable units. Modularizing your code helps you understand what your code is doing and makes your code more easily re-purposed for other projects.

5.2 Functions

We’ve been using functions for weeks now, but we haven’t dug deep into what makes a function a function.

5.2.1 Syntax

The first hurdle to learning to write a function in a programming language is figuring out what syntax you should use. In R, a function is defined using function() and has the following syntax:

rescale <- function(x){
  (x - min(x)) / (max(x) - min(x))
}

Let’s break this down into pieces:

Function Name: When you are reading the code above, the first thing you see is the name of the function (rescale). This name is chosen by the user, so we could have equally chosen to name the function bananna. However, that seems like a bad name since it doesn’t indicate what the goal of the function is. In general, we advocate to use verbs as function names where the verb describes what the function does.

Function Arguments: Inside the () after function is where the arguments (inputs) of the function are defined. Here we have one argument (x), but we will learn how to incorporate more than one argument.

Function Body: After the function(x) there is a { which starts the body of the function. The body of the function is everything contained between the opening { and the closing }.

Beginning & Ending the Function: Similar to other code styling principles we’ve learned, the guideline is for the opening { to be the last character on the line and for the closing } to be placed on its own line. These guidelines are to make your code easier to read / understand. The code below accomplishes the same task as above but is harder to read / understand what is happening:

rescale <- function(x){ (x - min(x)) / (max(x) - min(x)) }

5.2.2 Arguments & Parameters

Currently, the rescale() function has only one argument (x), but what if it had more? What are other arguments that we might want the function to use?

Well, suppose we had a vector with some missing values (NAs):

nums <- c(15:20, NA, 35:40, NA)

If we were to try and use our rescale() function on nums we would get:

rescale(nums)
 [1] NA NA NA NA NA NA NA NA NA NA NA NA NA NA

That doesn’t look right! What might be happening here?

When a vector has NA values, any mathematical operation (e.g., +, -) will result in an NA value. This is because R refuses to ignore the missing values (i.e., treat them as 0) unless the user specifies that is what they want. To specify that you want to remove the missing values, you need to use the na.rm argument:

mean(nums, na.rm = TRUE)
[1] 27.5

Let’s incorporate something like this into our function!

5.2.2.1 Removing Missing Values

Similar to the mean() function, we could incorporate an na.rm argument into our rescale() function. We could allow the user to specify if they wanted missing values to be removed (TRUE) or not (FALSE). Our calculations would then change based on what the user specified:

rescale <- function(x, na.rm){
  if(na.rm == TRUE){
    (x - min(x, na.rm = TRUE)) / (max(x, na.rm = TRUE) - min(x, na.rm = TRUE))
  } else {
    (x - min(x)) / (max(x) - min(x))
  }
}

To incorporate this argument we get to use our knowledge of control flow! Specifically, we can use if and else to define two possible options of what should happen based on the user’s input. Our rescale() function should now work for vectors with missing values!

rescale(nums, na.rm = TRUE)
 [1] 0.00 0.04 0.08 0.12 0.16 0.20   NA 0.80 0.84 0.88 0.92 0.96 1.00   NA

However, notice what happens when we don’t include an na.rm argument:

rescale(nums)
Error in `rescale()`:
! argument "na.rm" is missing, with no default

So, our function won’t run unless the user provides a value to the na.rm argument. This isn’t what the mean() function does! Indeed, you can run mean(nums) and get NA, which means that there is a default value for the na.rm argument that is used if the user doesn’t specify a value.

If you look at the help file for the mean() function (typing ?mean into your Console), you will see that by default na.rm is set to FALSE (na.rm = FALSE). We can incorporate something like this into our function!

rescale <- function(x, na.rm = FALSE){
  if(na.rm == TRUE){
    (x - min(x, na.rm = TRUE)) / (max(x, na.rm = TRUE) - min(x, na.rm = TRUE))
  } else {
    (x - min(x)) / (max(x) - min(x))
  }
}

Now, if the user doesn’t specify a value for na.rm it defaults to FALSE:

rescale(nums)
 [1] NA NA NA NA NA NA NA NA NA NA NA NA NA NA

5.2.2.2 Choosing a Rescale Method

Another addition we could make to our function is to allow the user to choose what rescaling method they want to use. Currently, our function uses min-max scaling, but there are other methods of scaling a vector of numbers. Arguably the most common scaling method is “normalization.” With normalization, the mean and standard deviation are used to rescale values.

Mathematically, each \(i^{th}\) value of a vector \(x\) is scaled based on this calculation:

\[\frac{x_i - mean(x)}{std(x)}\]

What if we add an argument into the rescale() function that allows the user to specify what method should be used to rescale?! It would make the most sense for the argument to accept a character input for the name of the scaling method (e.g. "min-max", "normalize"). Let’s see how we might incorporate this!

Before incorporating this option in to the function, let’s write out how many possible options there will be:

  1. "min-max" with na.rm = TRUE
  2. "min-max" with na.rm = FALSE
  3. "normalize" with na.rm = TRUE
  4. "normalize" with na.rm = FALSE

There are two things we should notice here. First, since there are four options of what can happen, we will need to make use of else if () statements. Second, each of these conditions requires two things to be true at the same time. Let’s learn how we can join relational statements together!

NoteJoining Relational Statements

When you have two relational statements (e.g., method =="normalize", na.rm == TRUE), there are two ways you can join them—an “and” or an “or.”

With an “and” (&) both statements must be TRUE simultaneously in order for the condition to be TRUE. If either statement is FALSE, then condition will return FALSE.

With an “or” (|) at least one statement must be TRUE for the condition to be TRUE. An or is less stringent and will only be FALSE if every statement is FALSE.

Relational Statement When is it TRUE?
x == 2 & y == 4 When both statements are TRUE
x == 2 | y == 4 When one statement is TRUE

Okay, now we’re ready to try this out. We know we need four total options, so we’re going to use one if (), two else if (), and one else. We’re also going to need to join two relational statements together for each of these options. Here we go!

rescale <- function(x, na.rm = FALSE, method){
  
  if (method == "min-max" & na.rm == TRUE){
    (x - min(x, na.rm = TRUE)) / (max(x, na.rm = TRUE) - min(x, na.rm = TRUE))
  } else if (method == "min-max" & na.rm == FALSE) {
    (x - min(x)) / (max(x) - min(x))
  } else if (method == "normalize" & na.rm == TRUE){
    (x - mean(x, na.rm = TRUE)) / sd(x, na.rm = TRUE)
  } else{
    (x - mean(x)) / sd(x)
  }
}

5.2.4 Explicit Returns

Now that we’ve explored different options for function inputs, let’s talk more about function outputs. A function output is often called a “return” because of its tie to a return() statement. You might wonder why you haven’t seen a return() statement in a function yet and the answer is that return() statements are a somewhat contentious topic among R programmers. We say “contentious” because many programmers have staunch views of using a return() statement, either for or against. We think you should be aware of these views so you can decide which option you prefer.

Option 1: Using an explicit return

When we say “explicit return” we mean that there is a return() statement built in to the body of the function. This would look something like this:

rescale <- function(x){
  x_scaled <- (x - min(x)) / (max(x) - min(x))
  return(x_scaled)
}

Notice how the results of the rescaling are saved into an object (x_scaled) and then the return() statement explicitly states that x_scaled should be returned (output). Here are a few pros and cons of this method!

✅ Pros of Explicit return()

  • Clarity and explicit intent
  • Consistency with other languages

⚠️ Cons of Explicit return()

  • More typing / slightly less idiomatic use of R
  • Can be overused in simple functions
  • Can complicate control flow unnecessarily

Option 2: Not using an explicit return / Using last value calculated

This option effectively removes the return() statement from the function. Instead of using a return() statement, the function relies on R’s behavior to return the last expression that was run.

rescale <- function(x){
  (x - min(x)) / (max(x) - min(x))
}

✅ Pros of Implicit Return (no return() statement)

  • Concise and idiomatic
  • Encourages “last expression” thinking

⚠️ Cons of Implicit Return

  • Less obvious for beginners
  • Harder to do early exits (i.e., your function ends early for certain inputs)

Choosing an Option

As an R programmer, you will need to decide for yourself which of these two options you prefer. It might be hard to know without trying what option you would prefer, so we encourage you to play around and try both!

The key is that you must be mindful to not mix and match methods between the two. Take for example a classic function writing mistake:

rescale <- function(x){
  x_scaled <- (x - min(x)) / (max(x) - min(x))
}

If we were to try and use this function to rescale a vector (e.g., rescale(y)) nothing would be output! That is because the rescaled values are saved into an object (x_scaled) but never returned!

NoteConsistency with code style guide

Technically, the code style guide we’ve been using takes an anti explicit return stance. If you want to hear more about their thinking, you can look at these examples.

5.3 Debugging

Sometimes function writing doesn’t go as smoothly as what we’ve walked you through. Sometimes your function outputs something unexpected or sometimes it doesn’t run at all. This can be very frustrating, but there are a few methods you can used to try and hunt down the problem.

5.3.1 print() Statements

Suppose our function wasn’t working as intended.

x <- 10:20
rescale(x, method = "min-max")
 [1] -1.5075567 -1.2060454 -0.9045340 -0.6030227 -0.3015113  0.0000000
 [7]  0.3015113  0.6030227  0.9045340  1.2060454  1.5075567

How do we know it’s not working as intended? Well, with min-max scaling, the largest value should be 1 and there should be no negative numbers.

One method for debugging is to add a print() statement into the body of the function. A print() statement does exactly what it sounds like, it prints the value of an object. For example, print(nums) would output 15, 16, 17, 18, 19, 20, NA, 35, 36, 37, 38, 39, 40, NA. Let’s see how this might look.

Here is the code for the broken function:

rescale <- function(x, na.rm = FALSE, method){
  
  if (method == "min-max" & na.rm == TRUE){
    (x - min(x, na.rm = TRUE)) / (max(x, na.rm = TRUE) - min(x, na.rm = TRUE))
  } else if (method == "minmax" & na.rm == FALSE) {
    (x - min(x)) / (max(x) - min(x))
  } else if (method == "normalize" & na.rm == TRUE){
    (x - mean(x, na.rm = TRUE)) / sd(x, na.rm = TRUE)
  } else{
    (x - mean(x)) / sd(x)
  }
}

You might have already spotted our error, but let’s suppose it was less obvious. Since it seems like the incorrect method is being used, we could add a print() statement at the beginning of the function to print out the value of method being input into the control flow.

rescale <- function(x, na.rm = FALSE, method){
  
  print(method)
  
  if (method == "min-max" & na.rm == TRUE){
    (x - min(x, na.rm = TRUE)) / (max(x, na.rm = TRUE) - min(x, na.rm = TRUE))
  } else if (method == "minmax" & na.rm == FALSE) {
    (x - min(x)) / (max(x) - min(x))
  } else if (method == "normalize" & na.rm == TRUE){
    (x - mean(x, na.rm = TRUE)) / sd(x, na.rm = TRUE)

  }

rescale(x, method = "min-max")

5.4 Input validation

When we write functions we know what type of inputs the function should use. However, someone else using our function might not know what the function expects. We will walk you through a series of input checks from simple to more complex,

5.4.1 Checking a Single Data Type

It might be obvious to us that the rescale() function should only be used with numeric values, but that might not be clear to someone else. We can prevent someone from giving our function a non-numeric input by adding a simple check at the beginning of the function:

rescale <- function(x, na.rm = FALSE, method){
  
  # check if x is a numeric vector
  stopifnot(is.numeric(x))
  
  if (method == "min-max" & na.rm == TRUE){
    (x - min(x, na.rm = TRUE)) / (max(x, na.rm = TRUE) - min(x, na.rm = TRUE))
  } else if (method == "min-max" & na.rm == FALSE) {
    (x - min(x)) / (max(x) - min(x))
  } else if (method == "normalize" & na.rm == TRUE){
    (x - mean(x, na.rm = TRUE)) / sd(x, na.rm = TRUE)
  } else{
    (x - mean(x)) / sd(x)
  }
}

The stopifnot() function takes one input, the condition you want to check. Here, we are checking if x is a numeric vector. The is.numeric() function will output two possible values, TRUE and FLASE. If the output is FALSE, then stopifnot() will terminate the function.

letters <- c("a", "b", "c", "d")
rescale(letters, method = "min-max")
Error in `rescale()`:
! is.numeric(x) is not TRUE

5.4.2 Checking Multiple Data Types

In a similar vein, we could check multiple data types still using the stopifnot() function. We also know the na.rm argument should be a logical value, and the method argument should be a character value. Let’s add these checks into the function!

rescale <- function(x, na.rm = FALSE, method){
  
  # check if all inputs are correct data type
  stopifnot(is.numeric(x), 
            is.logical(na.rm), 
            is.character(method)
            )
  
  if (method == "min-max" & na.rm == TRUE){
    (x - min(x, na.rm = TRUE)) / (max(x, na.rm = TRUE) - min(x, na.rm = TRUE))
  } else if (method == "min-max" & na.rm == FALSE) {
    (x - min(x)) / (max(x) - min(x))
  } else if (method == "normalize" & na.rm == TRUE){
    (x - mean(x, na.rm = TRUE)) / sd(x, na.rm = TRUE)
  } else{
    (x - mean(x)) / sd(x)
  }
}

You might wonder why we don’t have three stopifnot()s, one for each condition:

stopifnot(is.numeric(x)) 
stopifnot(is.logical(na.rm))
stopifnot(is.character(method))

That is a great question! When you give stopifnot() multiple inputs (e.g., stopifnot(is.numeric(x), is.logical(na.rm)), every input must be TRUE simultaneously for the function to run. If any of the inputs are FALSE the function will stop.

This is the same as joining these statements with an &:

stopifnot(is.numeric(x) & is.logical(na.rm) & is.character(method))

So, we use the three inputs (separated by commas) because (1) it requires the least amount of typing (instead of three stopifnot()s), and (2) it is easier to read (instead of using &s).

Let’s see if it works! Here are a few unexpected, but reasonable inputs:

rescale(10:20, na.rm = "true", method = "min-max")
Error in `rescale()`:
! is.logical(na.rm) is not TRUE
rescale(10:20, na.rm = TRUE, method = 2)
Error in `rescale()`:
! is.character(method) is not TRUE

5.4.3 Checking Object Size

Another common input check is to verify that the object(s) being input are the correct size. For example, our rescale() function expects x to be a vector with at least two values. Why? When calculating the standard deviation of x, if the vector has only one value the standard deviation is an NA:

x <- 5
sd(x)
[1] NA

To prevent this from happening, we can incorporate one additional check into our rescale() function. Specifically, we can check the length() of x and verify that it has more than 1 element:

rescale <- function(x, na.rm = FALSE, method){
  
  # check if all inputs are correct data type
  # and x has at least two elements
  stopifnot(is.numeric(x), 
            is.logical(na.rm), 
            is.character(method), 
            length(x) > 1
            )
  
  if (method == "min-max" & na.rm == TRUE){
    (x - min(x, na.rm = TRUE)) / (max(x, na.rm = TRUE) - min(x, na.rm = TRUE))
  } else if (method == "min-max" & na.rm == FALSE) {
    (x - min(x)) / (max(x) - min(x))
  } else if (method == "normalize" & na.rm == TRUE){
    (x - mean(x, na.rm = TRUE)) / sd(x, na.rm = TRUE)
  } else{
    (x - mean(x)) / sd(x)
  }
}

5.5 Environment, Scope, Name Masking & Dynamic Lookup

Now that we know how functions behave it’s time to learn some of the hidden details. Let’s start with the concept of a function’s environment.

5.5.1 Environment

A function has its own environment, a universe that gets opened when you run the function. A function’s environment is separate from the global environment, which is where all the objects you’ve created live (i.e., all the objects that would appear in the Environment tab). We need to first understand how these environments differ before we learn how they can relate to each other.

Let’s explore this with a simple function:

my_function <- function(){
  num = 67
  return(num)
}

Hopefully you’ve noticed something strange about this function—it has no arguments. We’ve written the function this way so we can explore the difference between a function environment and the global environment.

What would you expect this code to return?

my_function()

If you guessed 67 you are correct! When we call my_function() it first creates an object called num that has the value 67 attached to it, and then returns the value of num (67).

You might think that since my_function() creates an object num that you could use this object outside the function…

num
Error:
! object 'num' not found

but this object doesn’t exist in the global environment, it only exists in the function environment.

5.5.2 Name Masking

But what if there was an object in the global environment that had the same name as the object in the function environment?

num = 65

What do you expect this code will return?

my_function()

If you guessed 67 you are correct! This is an example of what is called name masking, where names (objects) defined inside of a function mask names (objects) defined outside of a function. In this example, num has the value of 65 outside the function, but inside the function num has a value of 67. Functions operate as a closed system where they rely only on the objects within the function environment, which is why my_function() returns 67 and not 65.

5.5.3 Dynamic Lookup

We just said “functions operate as a closed system,” but that is not always true. Functions try to operate as a closed system, but sometimes they need to ask for a little help. If we make a small modification to my_function(), you will see what we mean!

my_function <- function(){
  return(num)
}

Notice, we’ve eliminated the line of the function where num was created (num = 67). However, the function still returns the value of num. So, here’s our new system:

Now what do you expect this code will return?

my_function()

If you guessed 65 you are correct! When an object doesn’t exist in the function’s environment then it searches for that object in the global environment. If it can’t find an object in the global environment, the function will error out.

It is also important to note that because my_function() is relying on objects in the global environment, it is possible that the output can change. If we change the value of num in the global environment, then the output of my_function() will also change:

num = 13

my_function()
[1] 13

5.5.4 So what does this all mean?

This is a lot of technical details, so you might be wondering what the big picture is. Here are the three things you should know:

  1. Objects created in a function cannot be accessed in the global environment

  2. If an function can’t find an object it is supposed to use within its environment (the function environment), then it will search for that object in the global environment.

In general, we would recommend not using the same name for an object in a function and an object in your global environment. For example, you shouldn’t name your dataset data and then use a parameter data in your function.
